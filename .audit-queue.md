## CRITICAL (5)

### C1. Unhandled promise rejections in Worker/BroadcastWorker heartbeat
- **File**: src/worker.ts:228,344 | src/broadcast-worker.ts:235,368
- **Agents**: code-quality
- **Category**: error-handling
- **Description**: `void this.registerWorker()` in setInterval callback suppresses return but does not catch rejections. If registerWorker() fails, unhandled rejection can crash process.
- **Fix**: Add `.catch(() => {})` or `.catch(err => this.emit('error', err))` to registerWorker() calls
- **Effort**: small

### C2. Proxy validation gaps - NaN/Infinity/missing fields pass through
- **File**: src/proxy/routes.ts:127-146
- **Agents**: backend, security
- **Category**: validation
- **Description**: Proxy validates delay/priority/timeout/cost/attempts but NOT: jobId, ttl, ordering, deduplication, backoff, removeOnComplete, removeOnFail, lifo. NaN and Infinity values pass typeof number checks. Bulk endpoint (lines 194-205) has no per-job option validation at all.
- **Fix**: Validate all numeric options with Number.isFinite(), validate jobId/ordering/dedup structures, apply same validation to bulk endpoint.
- **Effort**: medium

### C3. Race condition in proxy queue cache - duplicate Queue creation
- **File**: src/proxy/routes.ts:80-105
- **Agents**: backend
- **Category**: race-condition
- **Description**: Two concurrent requests can both pass `!q` check and create two Queue instances for same name. Second overwrites first; first Queue is never closed, leaking connections and listeners.
- **Fix**: Track in-flight initializations with a Promise map, or use atomic set-if-absent pattern.
- **Effort**: medium

### C4. Poll loop promise rejection not fully handled on close
- **File**: src/worker.ts:1679 | src/broadcast-worker.ts:1666
- **Agents**: code-quality
- **Category**: error-handling
- **Description**: `void this.pollLoopPromise?.catch(() => {})` at close time - promise may not exist yet or may be in error state. Can leave unhandled rejection during shutdown.
- **Fix**: Store and track pollLoop promise reference consistently; ensure rejection is always consumed before nulling.
- **Effort**: medium

### C5. Producer cross-queue parent registration has no error handling
- **File**: src/producer.ts:299-301, 338-341
- **Agents**: backend, code-quality
- **Category**: error-handling
- **Description**: After addJob FCALL, cross-queue parent SADD is fire-and-forget. If it fails, child job exists but parent never sees it as a dependency - job hangs forever. No retry, no error propagation.
- **Fix**: Add try-catch with retry or throw, so caller knows registration failed.
- **Effort**: medium
